Overview 🌍

This project is a browser-based multiplayer spatial sandbox built with Node.js + Express, Socket.IO, Babylon.js, and mobile GPS + device orientation.

The system allows multiple users to inhabit a shared coordinate world while keeping raw GPS data private through an anchor-relative coordinate system.

PlayersWalk in the real world, see one another, and place objects.
ServerMaintains shared world state and distributes updates.
DaemonsAuthenticated helper services that create, destroy, or react to world objects.

Daemons are authenticated helper agents that can create and manage objects or respond to triggers.

Privacy Model 🔐

The project implements a privacy-preserving location system. Clients do not transmit raw GPS coordinates directly as world position.

Instead, each client chooses an anchor coordinate using this format:

lat,lon

Example value:

35.9940,-78.8986

The client calculates its offset from that anchor and transmits only:

  • anchor coordinate
  • relative X/Z distance from anchor
  • orientation/heading
  • object placement offsets

Anchor Worlds 🌐

Worlds are implicitly defined by anchor coordinates. Clients only see players using the same anchor and objects created under the same anchor.

Different anchors effectively create separate parallel worlds.

Server Responsibilities ⚙️

Main file: server/index.js

  • hosts static client files
  • manages Socket.IO connections
  • tracks connected clients
  • stores world objects
  • authenticates daemon clients
  • relays trigger events
  • distributes world updates

World State Storage

In the current version, world objects persist across disconnects and remain until explicitly deleted, wiped by the cleanup daemon, or lost on server restart. State is stored in memory only.

Client Responsibilities 🎮

Main files: public/index.html and public/main.js

  • obtains GPS locally
  • calculates anchor-relative position
  • transmits relative movement
  • renders other players
  • renders world objects
  • allows cube/object placement
  • allows proximity deletion
  • emits trigger events
  • manages Field Kit UI
  • renders a local orientation horizon

Field Kit UI 🧰

The Babylon GUI “Field Kit” provides in-game controls for anchor entry, anchor paste support, object placement, color/object controls, permission handling, and world interaction tools.

Anchor Entry System 📍

Anchor entry was simplified from two fields to a single input:

lat,lon

Example:

35.9940,-78.8986

The client also supports pasting coordinates directly from clipboard, which makes mobile use much easier.

Movement Model 🚶

The client calculates movement as:

relative position = GPS − anchor

The resulting offset is used for player rendering, world updates, object placement, and interaction logic. This allows cubes and objects to remain fixed in world space while players move.

Local Horizon Ring 🧭

The client renders a local orientation ring that is client-only, not synchronized to the server, not transmitted to other players, and follows local player position.

It serves as an orientation aid and heading reference. The ring lies in the horizontal plane around the player and includes small directional markers.

Object System 🧱

The project now supports a generic world object model. Previously the world only supported dropped cubes.

World objects can include:

  • position
  • anchor association
  • object type
  • visual properties
  • trigger properties
  • ownership
  • metadata

Trigger System ⚡

Objects may optionally contain trigger definitions such as proximity or interaction triggers.

player interaction
      ↓
client emits trigger
      ↓
server relays trigger
      ↓
daemon reacts

This enables daemon-driven mechanics without embedding all game logic directly into the main client.

Daemon Model 🤖

Daemons are special authenticated clients that can read world state, create objects, destroy objects, respond to triggers, and implement world mechanics.

Circle Daemon

File: public/circles.html
Auth cue: server/circle-auth.txt

The Circle Daemon can authenticate using its cue, create objects, react to triggers, spawn helper elements, and manage world mechanics.

Cleanup Daemon 🧹

File: public/cleanup.html
Auth cue: server/cleanup-auth.txt

The Cleanup Daemon allows controlled world resets and acts as a maintenance/admin tool. Wipe operations require deliberate confirmation.

Persistence Behavior 🧠

Objects persist across client disconnects and remain until manually deleted, removed by daemon, wiped, or lost on server restart.

This prevents objects from disappearing during brief mobile reconnects.

Major Changes In This Version 📜

  • Anchor-Relative Privacy Model: clients now compute and transmit relative coordinates instead of raw GPS world position.
  • Simplified Anchor Input: anchor entry changed to a single lat,lon field.
  • Clipboard Anchor Support: users can paste coordinates directly from maps or messages.
  • Local Horizon Ring: a client-side orientation ring was added and corrected to the horizontal plane.
  • Follow Behavior Correction: client follow logic now prioritizes live local position rather than server echo.
  • Expanded Object System: objects are now generalized rather than cube-only.
  • Circle Daemon Added: authenticated daemon for object creation and trigger experimentation.
  • Cleanup Daemon Added: authenticated world maintenance daemon.
  • Disconnect Object Deletion Removed: objects are no longer destroyed when player sockets disconnect.

Known Limitations ⚠️

  • No Disk Persistence: world state is not yet saved to disk.
  • Lightweight Authentication: daemon authentication is cue-based and not hardened.
  • GPS Accuracy Variance: movement quality depends on device GPS accuracy.
  • Early Object System: the architecture supports more object types than are currently polished.

Recommended Future Upgrades 🚀
